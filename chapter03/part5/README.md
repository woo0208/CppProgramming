# 실습과제 1

## 1. 함수를 사용하는 이유를 설명하라.
함수를 사용하는 주된 이유는 '코드의 모듈화(Modularity)'와 '재사용성(Reusability)'을 확보하기 위함입니다. 프로그램을 여러 개의 논리적인 단위(함수)로 분할하면 코드의 가독성이 향상되고, 동일한 기능이 필요할 때마다 코드를 중복 작성할 필요 없이 함수를 호출하기만 하면 됩니다. 이는 유지보수와 디버깅을 매우 효율적으로 만들어 주며, 객체 지향 프로그래밍에서는 클래스의 행위(Method)를 정의하는 핵심 수단이 됩니다.

## 2. 짧은 코드를 갖는 함수를 자주 호출하면 발생하는 함수의 오버헤드를 설명하라.

함수 호출 시에는 단순히 코드의 실행 위치만 바뀌는 것이 아닙니다. 실행 제어권 이동, 매개변수 전달을 위한 스택(Stack) 메모리 할당 및 인자 값 복사, 실행 후 돌아올 반환 주소(Return address) 저장, 그리고 실행 종료 후 스택 메모리 해제 등 일련의 내부 시스템 작업이 발생합니다. 이를 '함수 호출 오버헤드(Function Call Overhead)'라고 부릅니다. 
만약 함수의 내부 코드가 1~2줄로 매우 짧은데 이를 반복문 등을 통해 빈번하게 호출한다면, 실제 코드를 실행하는 시간보다 함수를 호출하고 복귀하는 데 걸리는 오버헤드 시간이 훨씬 더 커져 프로그램의 전반적인 실행 성능을 크게 저하시키게 됩니다.

## 3. 12페이지 예제에서 인라인함수가 치환되면 private 멤버 변수를 직접 접근하게 된다. 문법오류가 아닌가?
C++에서 클래스의 접근 지정자(public, private, protected)에 의한 접근 제어 및 권한 검사는 '컴파일 타임(Compile time)'에 해당 함수가 어느 소속인지를 기준으로 엄격하게 수행됩니다. 
인라인(inline) 함수가 호출부의 코드로 치환(Substitution)된다 하더라도, 컴파일러는 해당 코드가 본래 클래스의 멤버 함수라는 자격과 스코프(Scope)를 인지한 상태에서 처리합니다. 즉, 물리적으로는 호출 위치에 코드가 삽입되지만, 논리적으로는 여전히 멤버 함수로서 자신의 private 멤버 변수에 접근하는 정당한 권한을 행사하는 것이므로 문법적인 충돌이나 오류가 없습니다.

## 4. 인라인함수는 소스코드에서는 함수이지만 기계어로 번역되면 함수가 아니다. 이렇게 복잡하게 만든 이유를 설명하라. 다시 말해, 처음부터 함수를 안 쓰면 안되나요?
처음부터 함수를 사용하지 않고 메인 코드에 절차적으로 모든 코드를 풀어쓰게 되면, 앞서 1번에서 언급한 '모듈화', '가독성', '코드 재사용성'이라는 프로그래밍의 핵심 이점을 모두 포기해야 합니다. 특히 객체 지향 프로그래밍인 C++에서는 정보 은닉(Information Hiding)을 위해 멤버 변수를 private으로 캡슐화(Encapsulation)하고, 이를 읽고 쓰기 위한 Getter/Setter 등의 짧은 멤버 함수를 필연적으로 사용하게 됩니다.

인라인(inline) 함수는 바로 이 딜레마를 해결하기 위해 고안되었습니다. 프로그래머 입장에서는 소스 코드상에 명확한 '함수' 형태로 작성하여 캡슐화, 타입 안전성(Type safety), 가독성을 그대로 유지할 수 있습니다. 반면, 컴파일러는 이를 기계어로 번역할 때 함수 호출 과정을 생략하고 코드를 직접 삽입(Expansion)함으로써 2번에서 설명한 '함수 호출 오버헤드'를 완벽히 제거해 줍니다. 즉, '함수의 구조적 장점'과 '매크로(Macro) 수준의 실행 속도'를 모두 취하기 위해 제공되는 C++의 기능입니다.
