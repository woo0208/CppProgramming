# Q.강의노트에서 몰랐던 용어에 대하여 자세히 조사하시오

  ### 소스 레벨 호환성: 
  소스 코드를 수정하지 않고도, 새로운 버전의 라이브러리, 프레임워크, 컴파일러 환경 등에서 성공적으로 재컴파일이 가능한 상태, 
  즉 개발자가 기존 소스 코드를 건드릴 필요가 없을 때를 말합니다.

  ### 링크 레벨 호환성:
  소스 코드를 다시 컴파일할 필요 없이, 기존에 빌드된 목적 파일(.o)이나 실행 파일이
  새로운 버전의 라이브러리와 정상적으로 링크되고 실행되는 상태를 말합니다.
               
  ### 인라인 함수: 
  함수 호출 시 발생하는 오버헤드(문맥 교환, 파라미터 복사, 스택 프레임 생성 및 해제 등)를 제거하기 위해, 컴파일러가 함수 호출 위치에 함수의 실제 내부 구현 코드를 직접 복사   하여 치환(Expansion)하는 함수입니다.

  ### 디폴트 매개 변수: 
  함수를 선언할 때, 특정 매개 변수에 미리 기본값을 할당해 두는 문법적 기능힙니다.

  ### 제네릭 함수: 
  특정 데이터 타입에 종속되지 않고, 데이터 타입 자체를 매개 변수화(Type Parameter) 하여
  다양한 타입에 대해 동일한 로직을 수행할 수 있도록 일반화하여 구현한 함수입니다.

  ### 제네릭 클래스:
  클래스 내부에서 다룰 데이터 타입을 미리 지정하지 않고 클래스 외부에서 인스턴스를 생성할 때 결정하도록 일반화한 클래스입니다.

  ### 사이드 이펙트: 
  함수가 본연의 목적인 결과값을 산출하여 반환하는 것 외에 프로그램의 외부 상태를 변경하는 모든 행위를 말합니다.



# Q.알고리즘 설계와 코딩 중 무엇이 중요하고 그 이유는 무엇인가?

    알고리즘 설계가 더욱 중요하다 그 이유는 알고리즘 설계는 단순히 코드를 어떻게 잘 짜냐를 얘기하는 것이 아닌 문제 해결을 위한 핵심 논리와 
    데이터 처리 순서 등을 기획하는 단계이므로, 소프트웨어의 근본적인 성능을 결정짓는다.(시간복잡도 & 공간 복잡도 등) 
    코딩의 경우는 설계된 알고리즘을 특정 프로그래밍 언어의 문법에 맞춰 기계가 이해할 수 있도록  단순히 변역하는 과정에 불과하기 때문이다. 
    따라서 알고리즘 설계가 논리적으로 결함이 있거나 비효율적이라면, 코딩이 완벽하더라도 느리거나 잘못된 결과를 낼 수 있다.


# Q.클래스와 객체가 무엇인지 설명하시오.

    클래스는 사용자가 정의하여 사용하는 추상적인 자료형이자 설계도이다. 객체는 이 클래스를 기반으로 메모리에 할당되어 실제 데이터를 가지고 동작하는 실체(Instance)이다.
    예를 들어 현실 세계의 자동차를 Car라는 클래스로 정의할 때, 자동차의 상태를 나타내는 int speed, char color[] 등은 클래스 내부의 멤버 변수가 된다. 
    반면 객체는 이 Car 클래스 타입을 기반으로 메모리에 할당되어 생성된 구체적인 대상(예: Car myCar; 코드를 통해 생성된 myCar라는 고유한 실체)을 의미한다.


# Q.절차지향 프로그래밍 언어와 객체지향 프로그래밍 언어의 차이를 조사하라.

    절차지향 프로그래밍: 프로그램이 실행되는 순서와 흐름을 중심으로 설계하는 하향식(Top-down) 접근 방식이며 데이터와 데이터를 처리하는 함수가 분리되어 있다.
                    또한 실행속도가 빠르지만 코드의 규모가 커질수록 유지보수 및 재사용이 매우 어렵다는 단점이 존재하며 대표적인 언어로는 C언어가 있다.
                     
    객체지향 프로그래밍: 현실 세계의 사물을 객체로 모델링하고, 이 객체들 간의 메시지 전달 및 상호작용을 중심으로 프로그램을 구성하는 상향식(Bottom-up) 접근 방식이다.
                    데이터와 이를 처리하는 함수를 하나의 객체 안에 묶는 캡슐화, 상위 객체의 기능 및 변수를 사용하는 상속, 상속 받은 기능 및 변수를 재정의 하여 쓰는 
                    다형성등이 객체 지향의 특징이며 대표적인 언어로는 C++, Java, Python 등이 있다.



# Q.라이브러리가 무엇인지 자세히 설명하시오.

    라이브러리(Libray)란 개발자가 소프트웨어 개발 시 매우 빈번하게 사용되는 표준화된 함수, 클래스, 매크로 등의 코드를 하나로 모은 모음집이다. 
    입출력, 계산, 네트워크 통신 등 복잡하거나 반복적인 기능을 매번 개발자가 처음부터 구현할 필요 없이, 이미 검증되고 컴파일된 코드를 호출하여 
    재사용을 할 수 있도록 제공되는 편리한 소프트웨어 자원이다.


# Q.라이브러리가 목적파일 형태로 존재하는 이유를 설명하라.

    라이브러리를 소스 코드로 제공하면 프로그램을 빌드할 때마다 라이브러리 코드까지 전부 새로 컴파일 해야하므로 매우 큰 시스템 자원 및 시간이 낭비된다. 
    하지만 이미 컴파일이 완료된 목적 파일 형태로 제공하면, 개발자의 소스 코드만 컴파일한 뒤 링크만 별도로 수행하므로 빌드 속도가 매우 빨라진다.


# Q.링크과정이 필요한 이유를 설명하라.

    링크(Linking) 과정은 컴파일러가 만들어낸 여러 개의 목적 파일(.o)과 라이브러리 파일들을 결합하여, 운영체제가 적재할 수 있는
    하나의 단일 실행 파일(.exe 등)을 만드는 단계이다. 컴파일러는 각각의 소스 파일을 독립적으로 컴파일하는데 이때 A 파일에서 B 파일의 함수를 
    호출하거나 printf() 같은 외부 라이브러리를 호출할 경우,  컴파일러는 해당 함수의 실제 메모리 주소를 알 수 없으므로 주소값을 
    미확정 상태(외부 심볼)로 남겨둔 채 기계어(목적 파일)로 변환하게 된다. 
    
    만약 링크 과정이 없다면 이 외부 참조(External Reference)가 해결되지 않아 파편화된 코드들이 서로를 호출할 수 없고, 
    최종 실행 파일 빌드에 실패하게 된다.  따라서 링크 과정은 미확정된 심볼들의 실제 주소를 찾아 연결해 주고(Symbol Resolution), 
    흩어진 코드와 데이터 섹션들을 하나로 합쳐 메모리를 재배치(Relocation)해 주는 필수적인 역할을 한다.
